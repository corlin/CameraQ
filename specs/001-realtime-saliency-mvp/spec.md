# Feature Specification: Stage 2 Real-time Viewfinder MVP with Saliency

**Feature Branch**: `001-realtime-saliency-mvp`

**Created**: 2026-06-26

**Status**: Draft

**Input**: User description: "进入阶段 2 (实时取景器 MVP)，并在感知层引入显著性(Saliency)模型来解决“心形云”等非传统主体的识别问题。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实时摄像头推流感知 (Priority: P1)

用户打开应用后，系统能够调用设备摄像头获取实时视频流，并在屏幕上实时显示画面，同时在底层能够运行目标检测和显著性计算流水线。

**Why this priority**: 实时获取视频流并进行处理是 Stage 2 的基石，必须首先打通，否则无法提供“实时”反馈。

**Independent Test**: 用户启动应用后，能够看到流畅的后置摄像头实时画面，并能看到基本的检测识别框跟踪。

**Acceptance Scenarios**:

1. **Given** 应用已启动，**When** 用户授权相机权限，**Then** 屏幕应立即显示无明显卡顿的摄像头实时预览。
2. **Given** 正在预览实时画面，**When** 镜头前出现不断移动的物体，**Then** 系统底层处理管线能够每秒稳定输出分析结果，并且完全不造成预览画面的冻结或掉帧。

---

### User Story 2 - 非标准主体识别 (显著性检测) (Priority: P2)

当用户拍摄非标准的审美主体（例如一朵特殊形状的云、一束光斑、一片孤零零的树叶）时，系统不再仅仅依赖标准的实体类别库（如只找车或人），而是能够通过显著性（Saliency）分析，准确判断出画面中真正吸引人类视觉注意力的焦点。

**Why this priority**: 这是解决目前系统核心痛点（例如“心形云”无法识别而被下方车流劫持）的关键模块，是向“懂审美”跨越的第一步。

**Independent Test**: 在画面中仅有一朵形状奇特的云（非标准物体）且边缘有车辆或杂物时，系统能正确将“云”高亮为主目标。

**Acceptance Scenarios**:

1. **Given** 画面中存在标准物体（如车）和极具视觉吸引力的非标准物体（如高对比度的一朵云），**When** 系统进行主体裁定，**Then** 系统应将显著性最高的目标（云）选为主要关注点。
2. **Given** 系统识别到了非标准显著性主体，**When** 计算构图规则，**Then** 系统能够针对该显著性区域的包围盒，正确执行九宫格、居中等位置构图评判。

---

### User Story 3 - 取景器实时反馈交互 (Priority: P3)

用户在屏幕上能够实时看到轻量的构图辅助线（如九宫格），并且界面上只会在必要时“安静”地弹出一句简短的改进提示（如“向左移动一点”）。当用户按建议调整到位后，提示立刻消失。

**Why this priority**: 解决之前“拍后静态反馈”的体验割裂问题，打造真正的实时“伴随式拍摄助手”。

**Independent Test**: 在相机预览界面，当用户将主体置于画面左侧边缘时，系统立刻提示“建议向右微调”；当用户将镜头右移使主体落入三分线或中心后，提示自动变为“构图完美”并隐去。

**Acceptance Scenarios**:

1. **Given** 实时取景器正在运行，**When** 构图存在可量化的缺陷（如严重偏离重心或切边），**Then** 界面以无干扰的文字方式浮现最高优先级的修复建议。
2. **Given** 界面上正显示修复建议，**When** 用户移动机位解决了该问题，**Then** 警告提示自动消失或转为简短的正面反馈。

### Edge Cases

- 当画面中完全没有明显显著性区域（如拍摄一面纯白墙壁）时，系统如何处理？系统应保持静默，不强制生造一个主体，提示可置空。
- 手机/设备在快速移动时（Motion Blur），检测模型可能会失效或跳变。系统应引入追踪或缓冲机制，避免提示信息在屏幕上剧烈闪烁。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 能够合法请求摄像头权限并获取实时视频流数据。
- **FR-002**: System MUST 集成一种能够输出连续或离散注意力热图的 Saliency（显著性）检测引擎。
- **FR-003**: System MUST 具备一种融合决策层，当标准目标检测（如 YOLO）与显著性检测的目标冲突时，能基于显著度面积和分数做出唯一裁定。
- **FR-004**: System MUST 在处理管线中分离 UI 渲染与 AI 推理，确保无论 AI 推理耗时多久，取景器画面都保持极高帧率。
- **FR-005**: System MUST 在取景器界面上方叠加透明的构图辅助图层（如规则参考线和追踪框）。
- **FR-006**: System MUST 保证在任意时刻，辅助提示区域最多只显示一条最优先的单句建议文本。

### Key Entities *(include if feature involves data)*

- **SaliencyMap**: 显著性引擎输出的二维热度或灰度图，表示画面各像素点吸引眼球的概率。
- **FusedSubject**: 经过标准检测器与显著性模型融合评估后的最终目标对象，包含边框位置和置信度权重。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 取景器预览帧率（UI FPS）稳定在不低于 30 FPS。
- **SC-002**: AI 处理流水线的端到端单帧延迟应控制在 250 毫秒以内（即保证系统能每秒输出至少 4 次最新的构图决策）。
- **SC-003**: 在包含明显“非标准主体”（如奇特云朵、光影图案）的测试用例中，系统的主体判断准确率提升至 85% 以上。
- **SC-004**: 界面提示更新的抖动频率不得高于每秒 2 次，避免文字频繁闪烁造成视觉疲劳。

## Assumptions

- 假定在桌面或当前开发环境中，我们可以通过基于 Web（如 `getUserMedia`）或 OpenCV 调用摄像头来进行早期架构验证，暂时不要求第一版立刻打包为原生的 iOS/Android 离线应用。
- 假定设备算力足以支撑轻量级显著性网络（如 U^2-Net Lite）在低分辨率输入下的推理。
- 为了兼顾性能，假定 AI 分析频率可以低于摄像头刷新频率（如每 5 帧提取 1 帧进行显著性分析）。
